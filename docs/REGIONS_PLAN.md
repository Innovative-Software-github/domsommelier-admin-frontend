# Проектирование: регионы / города (Москва, Пермь)

Цель: ассортимент, мероприятия и прочая информация должны зависеть от города, который выбрал
пользователь. Документ описывает: (1) как бэкенд понимает выбранный город, (2) как фильтруется и
выводится информация, (3) как город выбирается на фронте. Затрагивает все три проекта:
`domsommelier-backend`, `domsommelier-frontend` (клиент), `domsommelier-admin-frontend` (админка).

---

## 1. Текущее состояние (что уже есть)

| Сущность | Привязка к городу сейчас | Вывод |
|----------|--------------------------|-------|
| `WineStore` | `city` + `district` (String, NOT NULL) | **Единственный носитель города.** Фильтр по city/district уже есть (`WineStoreFilterDto`), write под `@RequiresAdmin`. |
| `Event` | `city` (String, nullable) + связь `wineStore` | Уже city-aware, но город — свободная строка. |
| `Product` | **нет связи с городом** | `StorageHistory` = `product + amount + createdAt`, **без винотеки**. Склад глобальный. |
| `News` | нет | Общие для всех городов. |
| Клиент (фронт) | `city` Redux-слайс, `CitySelector`, `getUserCityFromHeaders` (хардкод «Москва») | `cities` пуст, город **не уходит** в запросы товаров/событий. |

**Главный вывод.** «Город» сейчас — свободная строка на `WineStore`/`Event`. Чтобы достоверно
ограничивать ассортимент по городу, не хватает двух вещей:
1. канонического справочника городов (иначе «москва»/«Москва»/«Moscow» разъедутся);
2. связи **товар → винотека (город)** — её сейчас нет вообще.

## 2. Ключевое архитектурное решение: что значит «товар доступен в городе»

Это развилка, от которой зависит объём доработок бэка. Два варианта:

### Вариант A — наличие по складам винотек (рекомендуется)
Новая таблица остатков по точкам: `ProductStock(product_id, wine_store_id, amount)`
(либо добавить `wine_store_id` в существующий `StorageHistory`).
«Ассортимент города X» = товары, у которых `amount > 0` хотя бы в одной винотеке города X.

- ➕ Точно, отражает реальные остатки; согласуется с самовывозом «через wineStore»
  (checkout уже привязан к винотеке).
- ➕ Бесплатно даёт «в каких точках города товар в наличии».
- ➖ Больше работы: миграция склада, управление остатками в админке.

### Вариант B — флаг доступности по региону (быстрый MVP)
Связь many-to-many `product_city(product_id, city_id)` — товар «продаётся в городе» без учёта остатка.

- ➕ Минимум кода, быстро.
- ➖ Не отражает реальные остатки; склад остаётся отдельной задачей; легко рассинхронизировать.

**Рекомендация:** целевой — **A**. Если нужно запуститься быстро — **B** как промежуточный шаг,
с последующей миграцией на A (контракт фронта при этом не меняется — см. §4).

> Решение по A/B нужно зафиксировать до старта — оно определяет схему БД и объём работ склада.

## 3. Справочник городов (общая основа для всех вариантов)

Вводим город как сущность первого класса — единый источник правды для селектора и фильтров.

```
City
  id        Long / UUID
  slug      String  UNIQUE   // 'moscow', 'perm' — стабильный ключ для API и URL
  name      String           // 'Москва', 'Пермь'
  active    boolean          // показывать ли в выборе
  sortOrder int
```

Связи:
- `WineStore.city` (String) → заменить на `WineStore.cityId` (FK → City). Миграция: по distinct
  значениям существующих строк создать City и проставить FK.
- `Event` — добавить `cityId` (FK), синхронно со строковым `city` на переходный период.
- В **Варианте A** город товара не хранится на товаре — он выводится через `винотека.city`.

Эндпоинт справочника (публичный GET):
```
GET /api/v1/cities                 → [{ id, slug, name }]  (только active)
```
Админ-CRUD (новый раздел в админке):
```
POST/PUT/DELETE /api/v1/admin/cities   @RequiresAdmin
```

## 4. Как бэкенд понимает, какой город выбран

**Принцип: город — это явный параметр запроса, а не часть JWT.**
Город — изменяемое предпочтение UI (пользователь переключает его в шапке), поэтому он не должен
«вшиваться» в токен. Контракт:

- Город передаётся как query-параметр **`city=<slug>`** на всех city-scoped эндпоинтах.
- Для залогиненного пользователя возможен дефолт на сервере: `Customer.defaultCity`
  (или `defaultWineStore.city`) — но он лишь подставляется, если параметр не пришёл.
- IP-geo (`x-vercel-ip-city`) — **только подсказка для первого выбора**, не источник правды.

Приоритет определения города (на фронте, см. §6), итог всегда уходит на бэк как `city=slug`:
```
явный выбор пользователя (cookie)  >  профиль (defaultCity)  >  IP-geo подсказка  >  дефолт 'moscow'
```

Серверная валидация (защита, а не только фильтр):
- На checkout: выбранная `wineStore` должна принадлежать переданному городу; каждая позиция
  заказа должна иметь остаток в этой винотеке (Вариант A) или товар должен быть доступен в городе
  (Вариант B). Иначе `409/422` с понятной ошибкой.
- `city` неизвестного slug → `400`.

## 5. Как выводится / фильтруется информация

Единый паттерн: добавить необязательный `city` к city-scoped запросам. Если `city` не задан —
поведение «как раньше» (все города), чтобы не ломать существующие вызовы.

| Эндпоинт | Изменение | Семантика при `city=perm` |
|----------|-----------|---------------------------|
| `POST /api/v1/products/filter` | + `?city=<slug>` | только товары, доступные в Перми (A: есть остаток в винотеке Перми; B: есть в `product_city`) |
| `GET /api/v1/products/{id}` | + `?city=<slug>` (опц.) | деталь + блок «наличие в городе»: список винотек Перми с остатком |
| `GET /api/v1/events/filter` | + `?city=<slug>` | мероприятия Перми |
| `GET /api/v1/wine-stores` | уже есть `city` фильтр | винотеки Перми |
| `GET /api/v1/news` | опц. `city` | общие + (опц.) таргетированные на город |
| `POST /.../checkout/{wineStoreId}` | валидация принадлежности города | см. §4 |

Реализация фильтра (Вариант A) — через JPA `Specification<Product>`: join `ProductStock` →
`WineStore` → фильтр по `city.slug` и `amount > 0`, `distinct`. Это ложится на текущий механизм
спецификаций (`BaseSpecification`).

Важно: ответы товара/события можно дополнить полем `availableCities: string[]` или
`inStockStoreIds: number[]` — чтобы фронт мог подсветить «нет в вашем городе» без второго запроса.

## 6. Как город выбирается на фронте (клиент, Next.js)

Уже есть: `store/city` (слайс `currentCity` + `cities`), `CitySelector`, серверная заглушка
`getUserCityFromHeaders`. Доработки:

1. **Контракт `ICity`.** Привести к `{ id, slug, name }` (сейчас `{ id, name }` с TODO).
2. **Источник списка.** `cities` грузить из `GET /api/v1/cities` (сейчас пусто). Сервис
   `services/city/requests.ts`.
3. **Первичный выбор города** (SSR, в `getCityInitialState`):
   `cookie 'city'` → профиль (`defaultCity`) → IP-geo (`x-vercel-ip-city` → матч по City) → `'moscow'`.
4. **Хранение выбора.** Писать slug в cookie `city` (доступна и на сервере, и в браузере).
   `customFetch` на серверной ветке уже умеет читать cookie (как делает с токеном) — по аналогии
   подставлять `city` в запросы; на клиенте брать из Redux/cookie.
5. **Проброс города в запросы.** Добавить `city` в `getFilteredProducts`, `getProductById`,
   события. Точка чтения — селектор `currentCitySelector`.
6. **Смена города.** При выборе в `CitySelector`:
   - обновить Redux + cookie;
   - перезапросить каталог/мероприятия/винотеки;
   - **провалидировать корзину**: если в новом городе часть позиций недоступна — показать модалку
     «эти товары недоступны в г. N» и предложить убрать их (корзина уже привязана к городу через
     винотеку самовывоза).
7. **Индикация.** Карточки/детали показывают «нет в вашем городе» по `availableCities`/наличию.

UX-замечание: город выбирается **до** добавления в корзину; основная винотека самовывоза должна
принадлежать выбранному городу (связать `CitySelector` с выбором винотеки в checkout).

## 7. Доработки в админке (domsommelier-admin-frontend)

Админка уже умеет CRUD винотек с полями city/district (см. `wineStoreForm`). Изменения:

1. **Новый раздел «Города»** (`features/cities`): CRUD справочника City (slug, name, active, sort).
   Меню-пункт + роуты + `api/cities/*`.
2. **Форма винотеки.** Поле «Город» — `Select` из справочника City вместо свободного `Input`
   (сейчас плейсхолдер `'москва'` строкой — это и есть риск рассинхрона). `district` оставить вводом
   или сделать зависимым списком.
3. **Форма мероприятия.** Поле «Город» — `Select` из City.
4. **Каталог/склад (Вариант A).** В форме товара — управление остатками по винотекам
   (`ProductStock`): таблица «винотека → количество». Это и есть механизм «положить товар в город».
5. **Фильтры списков.** В списках товаров/мероприятий добавить фильтр по городу (для удобства
   администратора), переиспользуя `Select` городов.

## 8. Сводка доработок по слоям

**Backend**
- `City` entity + `GET /api/v1/cities` + admin-CRUD.
- `WineStore.city` (String) → FK на City (+ миграция distinct-значений).
- `Event` — `cityId` (FK).
- **Вариант A:** `ProductStock(product, wineStore, amount)` (или `wine_store_id` в `StorageHistory`)
  + `Specification` фильтра по городу; **Вариант B:** таблица `product_city`.
- `city` query-параметр на `products/filter`, `products/{id}`, `events/filter`, (опц.) `news`.
- Серверная валидация города на checkout.
- (Опц.) поля ответа `availableCities` / `inStockStoreIds`.

**Frontend (клиент)**
- `ICity → { id, slug, name }`, `services/city/requests.ts` (`GET /cities`).
- Первичный выбор: cookie → профиль → IP-geo → дефолт; хранение в cookie.
- Проброс `city` в запросы каталога/деталей/мероприятий.
- Валидация корзины при смене города + индикация наличия.

**Admin**
- Раздел «Города» (CRUD).
- Город как `Select` в формах винотеки и мероприятия.
- (Вариант A) управление остатками по винотекам в форме товара.

## 9. Этапы внедрения

| Этап | Содержание | Результат |
|------|------------|-----------|
| **R0. Справочник** | `City` + `GET /cities`, миграция строковых городов, раздел «Города» в админке, City-`Select` в формах | Единый источник городов, без рассинхрона строк |
| **R1. Выбор на клиенте** | контракт `ICity`, загрузка `cities`, первичный выбор + cookie, `CitySelector` рабочий | Пользователь реально выбирает город, он сохраняется |
| **R2. Фильтрация вывода** | `city` в `events/filter` (готово по данным) и `wine-stores` (готово); проброс на фронте | Мероприятия и винотеки уже зависят от города |
| **R3. Ассортимент** | Вариант A: `ProductStock` + фильтр товаров по городу + управление остатками в админке (или B как времянка) | Каталог зависит от города |
| **R4. Валидация** | серверные проверки города на checkout, валидация корзины при смене города | Нельзя оформить недоступный в городе заказ |

> R2 даёт быстрый видимый эффект (мероприятия/винотеки) почти без схемных изменений — хороший
> первый поставляемый результат. R3 — самый объёмный (зависит от выбора A/B в §2).
